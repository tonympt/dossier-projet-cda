Réserver le stock : confirmCheckout

checkout/checkout.service.ts
async confirmCheckout(userId, orderId) {
  return this.prisma.$transaction(async (tx) => {
    const order = await tx.order.findUnique({
      where: { id: orderId },
      include: { items: { include: { tree: { include: { stock: true }}}}},
    });

    if (!order) throw new NotFoundException("Commande introuvable");
    if (order.userId !== userId)
      throw new ForbiddenException("Accès non autorisé");

    if (order.stockReservedAt ||
        order.paymentStatus === "PROCESSING")
      return { success: true };            // double-clic : rien à rejouer

    const issues = [];
    for (const item of order.items) {
      const available = item.tree.stock?.quantity ?? 0;
      if (available < item.quantity) issues.push({ treeId: item.treeId });
    }
    if (issues.length > 0) return { success: false, issues };

    for (const item of order.items)                        // décrément atomique
      await tx.treeStock.update({
        where: { treeId: item.treeId },
        data: { quantity: { decrement: item.quantity }},
      });
    await tx.order.update({
      where: { id: orderId },
      data: { stockReservedAt: new Date(), paymentStatus: "PROCESSING" },
    });
    return { success: true };
  });
}
  1. 1Transaction atomique$transaction : tout-ou-rien, rollback si erreur
  2. 2Contrôle d'accèsvérif propriétaire → 403 (OWASP A01 - Broken Access Control)
  3. 3Idempotencestock déjà réservé (double-clic) → on ne rejoue pas
  4. 4Valider avant d'écrire1 manque → rien n'est décrémenté
  5. 5Décrément atomique+ stockReservedAt + PROCESSING
  • Étape → back : réserver le stock
  • Pièce maîtresse → réserver le stock : atomique · sécurisé · idempotent, avant le paiement
  • Transaction → tout-ou-rien (rollback)
  • Contrôle d'accès → order.userId ≠ userId → 403 (OWASP A01)
  • Idempotence → stockReservedAt / PROCESSING → double-clic ne rejoue pas
  • Valider avant d'écrire → 1 manque → rien décrémenté
  • → Suite : le stock est réservé ; le paiement, lui, se joue via Stripe et son webhook

// payment.service.ts
async handleWebhook(signature, rawBody) {
  const event = this.stripe.webhooks.constructEvent(
    rawBody, signature, process.env.STRIPE_WEBHOOK_SECRET,
  );  // signature invalide -> 400

  const pi = event.data.object;
  switch (event.type) {
    case "payment_intent.succeeded":      return this.handleSuccess(pi);
    case "payment_intent.payment_failed": return this.handleFailed(pi);
    case "payment_intent.canceled":       return this.handleCanceled(pi);
  }
}

private async handleSuccess(pi) {
  const order = await prisma.order.findUnique({
    where: { id: +pi.metadata.orderId },
  });
  if (order.paymentIntentId !== pi.id) return;   // anti-spoofing
  if (order.paymentStatus === "PAID") return;    // idempotence

  await prisma.$transaction(async (tx) => {
    await tx.order.update({ where: { id: order.id }, data: {
      paymentStatus: "PAID", orderStatus: "CONFIRMED",
      invoiceNumber: await generateInvoiceNumber(tx),
    }});
  });
  await this.emailService.sendOrderConfirmationEmail(order);
}

  2. 2Signature HMACconstructEvent : seul Stripe peut signer (rawBody requis)
  3. 33 événementssucceeded / failed (restaure le stock) / canceled
  4. 4Sécurité du traitementanti-spoofing (paymentIntentId) + idempotence (déjà PAID)
  5. 5ConfirmationPAID + CONFIRMED + facture + email, en transaction

SQL équivalent (PostgreSQL)
BEGIN;

-- includes imbriqués = JOINs
SELECT o.*, oi.*, t.*, s.quantity
  FROM orders o
  JOIN order_items oi ON oi.order_id = o.id
  JOIN trees t        ON t.id = oi.tree_id
  JOIN stock s        ON s.tree_id = t.id
 WHERE o.id = $1;

-- decrement = UPDATE atomique
UPDATE stock
   SET quantity = quantity - $2
 WHERE tree_id = $3;

UPDATE orders
   SET payment_status = 'PROCESSING'
 WHERE id = $1;

COMMIT;
$transaction ↔ BEGIN … COMMIT include ↔ SELECT + JOINs decrement ↔ UPDATE atomique order.update ↔ UPDATE orders
  • Étape → données : la transaction, vue SQL
  • $transaction ↔ BEGIN … COMMIT/ROLLBACK
  • include ↔ JOINs · decrement ↔ UPDATE atomique (un par arbre, pas de race condition)

Redis : NoSQL clé / valeur, en mémoire

Base NoSQL clé / valeur, en mémoire (~0,1 ms) - trois usages concrets couvrent l'exigence NoSQL

Cache (cache-aside)

Miss → BDD → remplit le cache. Invalidation à l'écriture.

Pays 24 hCatégories 1 h
Anti-brute-force

Compteur d'échecs par email (incr + TTL).

≥ 5 échecs → bloqué 15 min
Blacklist JWT

Token blacklisté au logout, vérifié à chaque requête.

TTL = expiration restante

Le cache accroît les performances mais n'est pas critique : si Redis tombe, l'application continue de fonctionner (repli automatique sur la base de données).

  • Étape → données : le NoSQL
  • NoSQL en mémoire → ~0,1 ms, le bon outil pour ces 3 usages
  • Cache cache-aside → pays 24h · catégories 1h + invalidation → moins de requêtes SQL redondantes (éco-conception)
  • Anti-brute-force → 5 échecs/email → 15 min
  • Blacklist JWT → révoque un token stateless au logout

CSRF & contrôle d'accès

OWASP Top 10 : 2025

Contrôler qui peut faire quoi - de la requête forgée au risque n°1 OWASP

CSRF
Requête forgée · rattachée à A01
  • CORS restrictif - une seule origine autorisée (FRONTEND_URL)
  • Cookie SameSite=Lax - pas d'envoi du cookie en cross-site
  • Résultat : une requête forgée depuis un autre site est bloquée
Contrôle d'accès
OWASP A01 · Broken Access Control
  • JwtAuthGuard + RolesGuard - rôles USER / ADMIN
  • @Roles / @CurrentUser - déclaratif sur chaque route
  • Vérification propriétaire - order.userId === userId
  • CSRF → CORS restrictif (1 origine) + cookie SameSite=Lax → requête forgée bloquée
  • Contrôle d'accès (A01, risque n°1) → JwtAuthGuard + RolesGuard + ownership (order.userId)
  • Rappel → le serveur déclare, le navigateur applique (CORS/SameSite)
